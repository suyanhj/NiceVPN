# OpenVPN 管理系统宪法

## 核心原则

### 一、安全优先（不可妥协）

每一项功能都必须将安全性作为首要约束：

- 配置文件必须通过"硬件指纹 + OpenVPN `unique-id`"绑定到单一设备；服务端必须在底层直接拒绝多设备共用同一配置文件的连接请求。
- 证书吊销必须自动触发 CRL 更新，不允许存在任何需要手动干预的步骤。
- CA 操作必须要求密码验证，并向用户展示"更新 CA 将导致所有现有客户端失效"的强制提醒。
- 敏感操作（CA 更新、删除组、修改子网）在执行前必须要求用户进行二次确认。
- `.ovpn` 配置文件的下载链接必须是一次性的，且在 1 小时内过期。

**原则依据**: VPN 系统属于安全基础设施。设备绑定或证书管理的任何漏洞都将直接危及网络访问控制。

### 二、全图形化操作

所有管理任务必须可通过图形界面完成，用户不得需要执行任何 Linux 命令行操作。

- 每一个生命周期动作（创建、删除、启用、禁用、重启）必须有对应的 UI 控件。
- 防火墙规则排序必须支持拖拽；后端必须根据新顺序自动重建 `iptables` 规则。
- 配置文件在线编辑前必须自动备份原文件，方可允许原地修改。
- 首次运行向导必须强制完成全局 VPN 子网配置（含 CIDR 格式校验），系统方可进入正常运行状态。

**原则依据**: 本系统的核心价值在于消除命令行操作的复杂性。任何操作缺口都会迫使用户回退到命令行，直接破坏产品价值。

### 三、状态一致性

所有配置状态的变更在持久化前必须经过校验：

- 子网分配前必须检测是否与全局 VPN 网段及所有现有组网段存在重叠。
- 含有活跃用户的组不得修改其网段。
- 组内所有用户未删除前，不得删除该组。
- 防火墙规则在保存前必须校验端口范围、CIDR 格式及优先级冲突。
- 所有配置状态必须支持以 JSON 格式导出和导入，用于备份与灾难恢复。

**原则依据**: 子网重叠和陈旧的防火墙规则会造成无声的连接故障，在运行时极难诊断。

### 四、可观测性与可审计性

所有重要系统事件必须被记录并可视化呈现：

- 关键操作必须记录：操作人身份、时间戳、操作内容、执行结果。
- 审计日志必须是追加写入且防篡改的；禁止对已有日志条目进行原地修改。
- 仪表盘必须实时展示：在线设备数、流量统计、服务健康状态、已生效的防火墙规则数量。
- 以下场景必须触发告警：子网冲突、规则失效、服务宕机、证书在 7 天内到期。
- 钉钉机器人通知的使用范围仅限于分发限时 `.ovpn` 下载链接；不得将其作为系统内告警的替代方案。

**原则依据**: VPN 基础设施要求操作可追责、事件可快速发现。不可审计的变更会造成合规漏洞和排障困难。

### 五、封装复用，禁止重复造轮子

必须使用现有的、经过验证的工具，而非重新实现其逻辑：

- 证书操作（生成、吊销、续签）必须通过 Python subprocess 封装调用官方 EasyRSA 脚本来完成。
- 防火墙规则执行必须使用 `iptables` + `ipset`；禁止编写自定义的数据包过滤逻辑。
- 服务管理必须使用 systemd 或直接进程控制；禁止编写自定义的守护进程监控代码。
- 在编写工具类代码前，必须优先穷尽 Python 标准库和已批准的第三方包。

**原则依据**: 重新实现密码学或数据包过滤逻辑会引入正确性风险和维护负担，同时无法提供任何差异化价值。

### 六、语言与文档规范

本项目的所有沟通与文档输出必须使用中文：

- AI 助手（包括 Claude）在本项目中的所有回复必须使用中文。
- 所有项目文档（规格说明书、实现计划、任务清单、宪法、注释说明等）必须以中文撰写。
- 代码内注释应使用中文，尤其是对复杂逻辑（设备绑定流程、iptables 规则重建、EasyRSA 调用）的说明性注释。
- 代码本身的标识符（变量名、函数名、类名）使用英文，遵循 Python 社区惯例。
- 提交信息（git commit message）建议使用中文描述，也可使用"类型: 描述"的英文格式。

**原则依据**: 本项目的开发者和使用者均为中文母语用户；统一使用中文可降低沟通成本、减少理解歧义，并确保文档对目标受众保持可读性。

## 技术栈与兼容性标准

以下技术选型在本项目中已固定，未经正式宪法修订不得替换：

| 关注点 | 要求选型 | 约束说明 |
|---|---|---|
| 开发语言 | Python 3.8+ | 最低 3.8；新代码推荐 3.10+ |
| Web UI | NiceGUI | 需支持拖拽交互；不可替换为纯 HTML/JS 前端 |
| 后端接口 | FastAPI | 需异步支持；不接受 Flask/Django |
| 防火墙 | iptables + ipset | 不使用 nftables 或 ufw 抽象层 |
| 证书管理 | EasyRSA（官方脚本） | 不实现自定义 PKI |
| 存储 | JSON + 本地文本文件 | 不引入外部数据库（SQLite、PostgreSQL 等） |
| 消息推送 | 钉钉机器人 API | 仅用于分发 `.ovpn` 下载链接 |
| 目标操作系统 | CentOS / Ubuntu / Debian | 三者均需验证；Alpine 和 Windows 不在范围内 |
| OpenVPN 版本 | **2.7.0+**（当前最新稳定版） | 不为 2.7.0 以下版本提供兼容性处理；所有配置通过 `.conf` 配置文件管理 |

## 开发质量门禁

每项功能在宣告完成前，以下所有门禁必须全部通过：

1. **安全门禁**：设备绑定、证书生命周期、敏感操作二次确认流程已通过人工或自动化测试验证。
2. **一致性门禁**：子网重叠检测及规则校验路径已用冲突场景进行测试。
3. **图形化门禁**：所有操作已验证可在不使用任何命令行的情况下通过 UI 完成。
4. **审计门禁**：所有新操作均产生包含必要字段（操作人、时间戳、内容、结果）的日志条目。
5. **兼容性门禁**：功能已在至少一个 CentOS/Ubuntu/Debian 环境中验证通过。
6. **备份门禁**：任何新增配置状态已纳入 JSON 导出/导入的覆盖范围。

复杂逻辑（设备绑定流程、`iptables` 规则重建、EasyRSA 调用）必须包含充分的中文行内注释，使新开发者无需借助外部文档即可理解流程。

## 治理规范

本宪法凌驾于 OpenVPN 管理系统项目的所有其他开发约定之上。本文档与其他指导方针发生冲突时，以本宪法为准。

**修订程序**:

1. 以书面形式提出修订：说明受影响的原则、变更内容及修订理由。
2. 确定版本升级类型：删除或重新定义原则为 MAJOR；新增或实质性扩展为 MINOR；澄清、措辞或排版修正为 PATCH。
3. 修订生效后更新 `LAST_AMENDED_DATE` 和 `CONSTITUTION_VERSION`。
4. 按一致性传播清单将变更同步至所有依赖模板。

**合规审查**: 每份功能实现计划（`plan.md`）必须包含"宪法核查"章节，确认与六条核心原则的对齐情况。任何违例必须在"复杂性追踪"表格中注明并提供充分说明。

**版本策略**: 采用语义化版本 — MAJOR.MINOR.PATCH。任何治理变更在版本行更新前不生效。
